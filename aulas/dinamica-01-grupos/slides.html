O que vocês podem usar

Somente o que vimos em aula — sem pesquisar estruturas novas!

Liberado

  • ✓ Variáveis: inteiro, real, caracter, logico
  • ✓ leia() e escreva()
  • ✓ Operadores aritméticos e relacionais
  • ✓ Álgebra booleana: e, ou, nao
  • ✓ Estrutura se / senao se / senao
  • ✓ Estrutura escolha / caso / outrocaso
  • ! Aninhamento de condicionais

Proibido

  • ✗ enquanto — laço de repetição
  • ✗ para — laço de repetição
  • ✗ repita / ate — laço de repetição
  • ✗ Funções e procedimentos personalizados
  • ✗ Vetores e matrizes
warning Usar qualquer estrutura proibida desconta 2 pontos na nota.

Grupo 1 — Detalhamento

🍔 Cardápio Digital

description Briefing do Cliente

Minha lanchonete precisa de um sistema simples. O atendente digita o número do pedido (1 a 5) e o programa mostra o nome do item, o preço e uma mensagem de confirmação. Número inválido exibe uma mensagem de erro amigável.

Itens do cardápio

  • 1 X-Burguer — R$ 18,00
  • 2 X-Frango — R$ 16,00
  • 3 Batata Frita — R$ 12,00
  • 4 Refrigerante — R$ 7,00
  • 5 Suco Natural — R$ 9,00

Critérios de aceite

  • ✓ Cada número exibe nome e preço corretos
  • ✓ Número inválido → mensagem de erro
  • ✓ Textos definidos pelo UX Writer
  • ! Obrigatório: escolha/caso
  • ✓ MD do projeto entregue com testes
Grupo 2 — Detalhamento

📋 Boletim Escolar

description Briefing do Cliente

Sou coordenador da escola. Quero um programa que receba o nome do aluno e sua média final (0 a 10), exiba a situação (aprovado, recuperação ou reprovado) e o conceito de A até D.

Regras de negócio

  • A Média ≥ 9.0 — Aprovado / Conceito A
  • B Média ≥ 7.0 — Aprovado / Conceito B
  • C Média ≥ 5.0 — Aprovado / Conceito C
  • D Média ≥ 3.0 — Recuperação / Conceito D
  • ✗ Média < 3.0 — Reprovado

Critérios de aceite

  • ✓ Nome do aluno aparece na saída
  • ✓ Situação e conceito corretos por faixa
  • ✓ Testar com: 9.5 / 7.0 / 5.0 / 3.0 / 1.0
  • ! Combinar se/senão com escolha/caso
Grupo 3 — Detalhamento

💪 IMC Fitness

description Briefing do Cliente

Sou personal trainer. Preciso de um app que receba o nome, peso (kg) e altura (m), calcule o IMC com a fórmula peso / (altura * altura), exiba a classificação e uma mensagem motivacional diferente para cada faixa.

Tabela de classificação

  • ↓ IMC < 18.5 — Abaixo do peso
  • ✓ IMC 18.5 – 24.9 — Peso normal
  • ! IMC 25.0 – 29.9 — Sobrepeso
  • ↑ IMC ≥ 30.0 — Obesidade

Critérios de aceite

  • ✓ Nome aparece na saudação inicial
  • ✓ IMC exibido com 2 casas decimais
  • ✓ Classificação e mensagem por faixa (UX)
  • ! Usar se/senão aninhado
Grupo 4 — Detalhamento

🏧 Caixa Eletrônico

description Briefing do Cliente

Sou gerente de banco. Preciso de um sistema onde o cliente informa o nome e o saldo atual, depois escolhe uma operação: 1-Consultar, 2-Depositar ou 3-Sacar. O sistema executa a operação e exibe o novo saldo.

Regras de negócio

  • 1 Consultar → exibe saldo atual
  • 2 Depositar → pede valor, soma ao saldo
  • 3 Sacar → pede valor, valida se há saldo suficiente
  • ✗ Saque com saldo insuficiente → mensagem de erro
  • ✗ Opção inválida → mensagem amigável

Critérios de aceite

  • ✓ Nome do cliente na saudação
  • ✓ Saldo atualizado exibido após operação
  • ✓ Saque inválido não altera o saldo
  • ! escolha para opção + se/senão para validação
Grupo 5 — Detalhamento

📐 Classificador de Triângulo

description Briefing do Cliente

Sou professor de geometria. Preciso de um programa que receba os 3 lados de um triângulo, verifique se eles formam um triângulo válido e, se sim, classifique se é Equilátero, Isósceles ou Escaleno.

Regras matemáticas

  • ✓ Válido se: a < b+c e b < a+c e c < a+b
  • E Equilátero: a == b e b == c
  • I Isósceles: apenas 2 lados iguais
  • S Escaleno: todos os lados diferentes

Critérios de aceite

  • ✓ Lados inválidos → mensagem de erro clara
  • ✓ Classificação correta para cada tipo
  • ✓ Testar: (3,3,3) / (3,3,5) / (3,4,5) / (1,2,10)
  • ! Usar e / ou obrigatório na validação

Para o Dev

Dicas de Código

Estrutura base em Portugol

programa { funcao inicio() { // 1. Declare TODAS as variáveis primeiro inteiro opcao real saldo, valor // 2. Leia as entradas escreva("Saldo atual: R$ ") leia(saldo) escreva("Escolha (1-3): ") leia(opcao) // 3. Tome a decisão escolha(opcao) { caso 1: escreva("Saldo: R$ ", saldo) caso 3: se(valor <= saldo) { saldo = saldo - valor } outrocaso: escreva("Opção inválida") } } }
lightbulb Declare todas as variáveis antes do primeiro leia().
lightbulb Use real para valores com vírgula (preço, IMC, saldo).
warning escolha só aceita inteiro ou caracter na expressão — não use real.
bug_report Sempre teste o caso limite: o que acontece com entrada inválida?
Para o QA

Como montar seus Testes

Crie a tabela de testes antes mesmo do código ficar pronto — é trabalho em paralelo!

# Entrada Saída esperada Resultado
CT01Valor válido (caso feliz)Resposta correta esperada— a testar
CT02Outro valor válidoResposta correta esperada— a testar
CT03Valor limite (borda)Resposta do limite— a testar
CT04Valor inválidoMensagem de erro amigável— a testar
CT05Caso extremo / negativoTratamento correto— a testar
info Após rodar: preencha cada linha com OK ou FALHA + o que saiu errado.
Para o UX Writer

Textos que fazem diferença

Textos ruins

  • ✗ "Digite opcao:"
  • ✗ "Erro."
  • ✗ "ok"
  • ✗ "resultado = 25"
  • ✗ Sem saudação ao iniciar

Textos bons

  • ✓ "Bem-vindo, [nome]!"
  • ✓ "Escolha um item (1 a 5): "
  • ✓ "Opção inválida. Tente entre 1 e 5."
  • ✓ "Pedido confirmado! Bom apetite!"
  • ✓ "Seu IMC é 24.5 — Peso normal!"
chat Entregue todos os textos prontos ao Dev antes de ele começar a codificar.
